# GAR Ops

## 1. 운영 목표

- 수집 파이프라인이 평일 업무시간 동안 안정적으로 동작할 것
- 중복 알림과 무한 재시도를 방지할 것
- 실패 원인을 로그와 메트릭으로 바로 추적할 수 있을 것

## 2. 스케줄

| 작업 | 주기 | 비고 |
|------|------|------|
| 공고 수집 | 평일 매시간 08:00~18:00 KST | `withoutOverlapping` |
| 마감 임박 알림 | 매일 09:00 KST | D-3, D-1 |
| 일일 다이제스트 | 매일 18:00 KST | 점수 50~69 포함 |
| 결과 추적 | 매일 10:00 KST | 고도화 단계 |

## 3. 큐 권장 분리

- `collect`
- `documents`
- `analysis`
- `matching`
- `notifications`
- `estimates`

## 4. 재시도 정책

| 작업 | 재시도 | 전략 |
|------|--------|------|
| API 수집 | 3회 | 지수 백오프 |
| 첨부 다운로드 | 3회 | 지수 백오프 |
| 문서 파싱 | 2회 | 동일 파서 1회, 대체 처리 1회 |
| AI 분석 | 2회 | 동일 모델 1회 재시도 |
| 알림 발송 | 채널별 3회 | 채널 SDK 기준 |

## 5. Idempotency

- 수집은 `bid_ntce_no + bid_ntce_ord`
- 첨부는 `bid_id + normalized_url`
- 알림은 `dedupe_key`
- 견적은 `company_id + bid_id + price_table_version`

## 6. 저장소 정책

- 첨부 저장 경로: `storage/app/bid-documents/{bid_id}/`
- 원문 응답 fixture: `storage/app/g2b-fixtures/`
- 첨부와 분석 원문은 최소 180일 보존
- 개인식별정보는 필요 최소한만 저장

## 7. 관측성

### 핵심 메트릭

- 시간당 수집 공고 수
- 필터 통과율
- 첨부 다운로드 성공률
- 파서별 성공률
- AI 분석 성공률
- 매칭 계산 시간
- 알림 발송 성공률

### 알림 기준

- 수집 2회 연속 실패
- 분석 성공률 80% 미만
- 알림 발송 실패율 10% 초과
- 큐 적체 15분 초과

## 8. 보안

- G2B 인증키와 AI 키는 환경변수만 사용
- 첨부파일은 실행 금지 저장소에 보관
- 업로드/다운로드 경로 탐색 취약점 방지
- 운영 로그에 원문 첨부 전체 내용을 남기지 않음

## 9. 백업

- DB 일일 백업
- 첨부 스토리지 주간 백업
- 가격표와 견적 데이터는 복구 우선순위 상위

## 10. 운영 전 체크리스트

- 실제 스케줄 타임존 확인
- 큐 워커 개수 결정
- 실패 알림 채널 설정
- 파일 스토리지 용량 예측
- 로그 보존 정책 확정
